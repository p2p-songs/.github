## Scope

This pass audits player P-4: the persistence port, memory and Dexie adapters,
library/playlists/installed-addons/settings/queue repositories, secret-bearing
addon records, queue hydration, and cross-addon catalog fan-out. It also
rechecks the A-008 failure-isolation changes shared by metadata and stream
providers. Verification included source and architecture inspection, built-
output adversarial probes, 277 tests across player/addons/SDK/protocol,
typechecks in all three TypeScript workspaces, and all six contract lenses.
Future P-5 app-shell wiring, UI, service-worker policy, and sync are not treated
as current-scope defects.

## Overall verdict

P-4 does not pass yet. The load-bearing rule that resolved media never reaches
durable storage is correctly enforced through both store adapters, configured
addon URLs are isolated and redacted, and normal catalog fan-out correctly
merges/deduplicates while isolating cooperative provider failures. Three
implemented-scope gaps remain: the shared deadline does not actually settle
against a task that ignores abort, concurrent playlist mutations lose updates,
and the explicitly promised play-history persistence is absent despite P-4
being marked done. A-008's ordinary fetch-abort paths are improved, but its
strong bounded-completion claim is therefore not confirmed.

## Findings

### [MEDIUM] The shared provider deadline still waits forever when abort is ignored

- **Category:** system-design
- **Repo / file / line:** `player/src/core/addon/fan-out.ts:22-45`; `player/src/core/addon/collection.ts:105-177`; `player/src/core/addon/stream-resolver.ts:78-142`
- **Reference:** Checklist §8 “Failure is bounded”; Contract §2a addon-outage/partial-results journey; ARCHITECTURE §5 failure domains; A-008 second finding
- **Finding:** `askBounded` starts a timer that aborts a child signal, but still directly awaits `task(child.signal)`. The timeout outcome is produced only if the task reacts by rejecting. A built-output probe passed a task that returns a never-settling promise and ignores the signal; racing `askBounded(..., 10)` against a 50 ms observer returned `still-pending`. The helper's statement that it “never rejects” is true, but the stronger documented promise that it settles within `providerTimeoutMs` is false.
- **Why it matters:** An injected transport, browser/network implementation, or addon request wrapper that fails to observe abort can still wedge stream resolution, metadata fallback, and search forever. Healthy co-providers remain trapped behind `Promise.all`, recreating the dead-play/spinner failure A-008 was intended to close.
- **Suggested fix:** Race the task against an independently settling timeout/outer-abort promise, abort the child for cleanup, and consume any late task rejection so it cannot become unhandled. Add a test whose task deliberately ignores its signal, plus late-resolve/late-reject tests.
- **Verdict:** CONFIRMED

### [MEDIUM] Concurrent playlist edits overwrite one another

- **Category:** implementation-quality
- **Repo / file / line:** `player/src/core/persistence/repository.ts:92-113`; `player/src/core/persistence/store.ts:9-15`; `player/src/core/persistence/dexie-store.ts:39-57`
- **Reference:** ARCHITECTURE §6 durable playlists; Contract §2a recovery without losing work; not otherwise covered by an existing atomicity invariant — flagging as new
- **Finding:** Every playlist edit is an unprotected read-modify-write across two separate store calls. Two mutations can read the same prior record and then each `put` a replacement; the last write silently discards the first. A built-output probe ran two `addToPlaylist` calls concurrently against a store with delayed reads/writes. Adding `A` and `B` to an empty playlist produced only `["B"]`.
- **Why it matters:** Two rapid UI actions, overlapping app callbacks, or future sync/local activity can silently lose a user's playlist edit. The API exposes async mutation methods without serialization or conflict detection, so callers have no indication that an accepted add vanished.
- **Suggested fix:** Put mutation behind an atomic store operation/transaction (Dexie read+write transaction) or serialize mutations per playlist, and add concurrent add/add and add/remove tests. The persistence port needs to express the chosen atomic primitive rather than composing `get` and `put` in the repository.
- **Verdict:** CONFIRMED

### [MEDIUM] P-4 is marked complete without the promised play-history persistence

- **Category:** technical-soundness
- **Repo / file / line:** `player/docs/ARCHITECTURE.md:479,948,984-1004`; `player/src/core/persistence/schema.ts:11-18`; `player/src/core/persistence/repository.ts:31-219`
- **Reference:** ARCHITECTURE §6 and §10 P-4 deliverable; Plan §7 storage decision
- **Finding:** P-4's declared deliverable and durable-library table explicitly include play history, but `COLLECTIONS` has only library, playlists, addons, settings, and queue. There is no history record type or repository API, while the phase is marked “DONE.” Unlike engine wiring, history is not identified as a P-5 deferral.
- **Why it matters:** The persistence layer cannot record or restore listening history, so a user-facing history/recently-played flow and the later listening-state sync contract have no durable source to build on. Declaring the data layer complete hides a missing schema/API until UI or sync work depends on it.
- **Suggested fix:** Add an identity-only play-history collection and repository operations with timestamps/retention semantics, or explicitly narrow the P-4 contract and document where history will land before calling the phase complete.
- **Verdict:** CONFIRMED

## Six-lens summary

- **Technical soundness:** Resolved stream URLs are stripped before persistence and hydrate to idle through MemoryStore and Dexie. Play history, an explicit P-4 deliverable, is missing.
- **Legal validity:** No finding. The audited slice stores no media, introduces no debrid account, keeps addon installation user-directed, and does not weaken the reference addons' source/license rules.
- **Overall implementation quality:** The store port is small and the adapters are well tested, but playlist read-modify-write is non-atomic and loses concurrent edits.
- **System design and operations:** Provider isolation works when transports honor abort; the shared helper is not a hard deadline against an uncooperative task.
- **UI/UX and accessibility:** No P-5 UI exists, so no separate accessibility finding applies. Missing history and lost playlist edits are recorded at the data boundary because they deterministically remove future user-visible state.
- **End-to-end user value and delight:** Reload restores queue identity without bearer media and search survives ordinary partial outages. A wedged provider can still create an indefinite wait, and overlapping playlist actions can silently discard work.
