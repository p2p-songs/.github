## Scope

This pass audits player P-3: addon URL construction, HTTP/JSON transport,
manifest/resource schema validation, installed-addon collection, the real
multi-addon stream resolver, provider health/backoff, integration with the P-1
scheduler/engine, and the live loopback `musicmeta`/`stream-legal` end-to-end
test. It also re-audits both A-007 fixes. The finding-calibration gates were
applied: only reproduced failures in the implemented P-3 boundary are counted;
future UI/query-policy, persistence, deployment topology, and optional
optimizations are not findings. Verification included source/package inspection,
built-output adversarial probes, 216 tests across player/addons/SDK/protocol,
typecheck/build in all three workspaces, repository/issue/PR state, and all six
contract lenses.

## Overall verdict

P-3 does not pass yet because addon failure isolation is incomplete in two
implemented primary paths. Metadata lookup stops at the first matching provider
that throws, so a down addon prevents a healthy fallback from contributing.
Stream fan-out uses unbounded `Promise.allSettled` with a default transport that
has no deadline, so one hung provider blocks a valid result already returned by
another provider forever. The rest of the slice is sound: configured URLs are
HTTPS-only outside loopback and never logged/cached, every response is schema-
validated, `/stream` remains a retry-free memory-only scheduler command,
provider backoff distinguishes reachable empty results from addon-wide faults,
no addon is bundled at runtime, and the live happy path works. Both A-007 fixes
are confirmed.

## Findings

### [MEDIUM] Metadata fallback stops when the first provider is unavailable

- **Category:** system-design
- **Repo / file / line:** `player/src/core/addon/collection.ts:56-68`; `player/src/core/addon/client.ts:121-124`; `player/src/core/addon/http.ts:76-103`
- **Reference:** Protocol §6 (“one addon failing never blocks”); Contract §2a addon-outage/partial-results journey; ARCHITECTURE §3 failure isolation
- **Finding:** `AddonCollection.getMeta` iterates matching providers, but it awaits each `getMeta` without isolating `AddonUnreachableError` or `AddonProtocolError`. The first exception exits the collection method and no later provider is queried. A built-output probe installed two matching meta addons, made the first throw a network failure, and configured the second to return valid metadata; `getMeta` threw `AddonUnreachableError`, and the recorded requests confirmed the healthy provider was never called.
- **Why it matters:** Installing an additional or temporarily unreliable metadata addon can make discovery worse: it shadows every healthy provider after it in installation order. Album/artist/track detail becomes unavailable even though the system has a valid answer, violating the addon protocol's fault-isolation value.
- **Suggested fix:** Isolate addon-wide failures per provider and continue to the next capable addon; preserve cancellation as cancellation, and surface aggregate failure only when no provider was reachable. Reuse provider-health/backoff semantics or a metadata-plane equivalent. Add down-first/healthy-second, malformed-first/healthy-second, empty-first/healthy-second, and abort tests.
- **Verdict:** CONFIRMED

### [MEDIUM] One hung stream addon blocks healthy providers indefinitely

- **Category:** end-to-end-user-value
- **Repo / file / line:** `player/src/core/addon/stream-resolver.ts:59-89`; `player/src/core/addon/http.ts:111-123`
- **Reference:** Contract §2a addon-outage/partial-results journey; ARCHITECTURE §4b/§5 failure domains and provider backoff
- **Finding:** Stream providers are awaited with `Promise.allSettled`, and `defaultHttpGet` forwards only the scheduler's cancellation signal; it establishes no per-provider deadline. `Promise.allSettled` cannot return partial results while any request remains pending. A built-output probe used one provider whose `/stream` promise never settled and one provider that immediately returned a valid HTTPS stream; after 25 ms the resolver was still pending rather than returning the healthy stream or classifying/backing off the hung provider.
- **Why it matters:** A single installed addon that accepts a connection but stalls can leave playback in `resolving` forever. Healthy providers, ranked fallback, the engine breaker, and provider backoff never get a chance to act because resolution never completes. This is a real spinner/dead-play condition at the new external boundary.
- **Suggested fix:** Give each provider request a bounded, abortable deadline and classify timeout as addon-wide unreachability/backoff. Either wait only for that finite provider window or return useful partial results under an explicitly bounded aggregation policy. Preserve the scheduler's outer abort and add hung+healthy, all-hung, timeout-backoff, and skip-during-timeout tests.
- **Verdict:** CONFIRMED

## Six-lens summary

- **Technical soundness:** Response validation, URL encoding, HTTPS/loopback transport policy, command-plane separation, A-007 stamp gating, and empty-queue selection are confirmed. Changes are required only for the two reproduced failure-isolation gaps.
- **Legal validity:** No finding. The player remains addon-neutral, bundles no runtime source, stores no media, and this slice adds no debrid account/content handling. Existing SDK/source legal invariants remain intact.
- **Overall implementation quality:** All 216 tests/typechecks/builds pass and the live HTTP fixture is valuable. Coverage omits down-first metadata fallback and a provider that hangs without honoring cancellation/deadline.
- **System design and operations:** Metadata provider failures are not isolated, and stream aggregation has no bounded completion. Provider exponential backoff works for failures that actually settle.
- **UI/UX and accessibility:** No UI is implemented, so no separate accessibility finding applies. The two engine/client failures are recorded because they deterministically produce future dead discovery/resolving states rather than actionable outcomes.
- **End-to-end user value and delight:** The happy-path album metadata → recording ID → resolved HTTPS URL → playing journey passes over real loopback HTTP. Partial-provider failure and hung-provider recovery do not.
