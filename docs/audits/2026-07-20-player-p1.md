## Scope

This pass audits the complete player P-1 headless engine—stable-ID queue,
playback reducer FSM, resolution/prefetch scheduler, fake resolver/audio backend,
orchestrator, fallback, breaker, and race tests—and re-audits all six A-006
fixes in the SDK/reference addons. The review traces queue creation/mutation,
resolve/load/play/skip/retry/prefetch, stale async completion, stream freshness,
and failure recovery. Later browser audio,
real addon client, persistence, UI/PWA, and accounts remain unimplemented and
are not failed as missing. Verification included source/package inspection,
built-output adversarial probes, 166 tests across the player/addons/SDK/protocol,
typecheck/build in all three workspaces, and all six contract lenses.

## Overall verdict

P-1 does not pass yet. The architecture's most important race invariant is only
enforced in the playback FSM, not at the queue-resolution commit: a superseded
resolve can land late, overwrite the newer `QueueItem.resolution` bearer URL,
and be reused on a later play. One medium queue defect remains: adding the first
item through `append` leaves the queue unplayable. A-006's configured
method no-store, meta request identity, license evidence, artist matching, and
outage semantics fixes are confirmed.

## Findings

### [HIGH] Superseded resolution results overwrite newer queue state

- **Category:** technical-soundness
- **Repo / file / line:** `player/src/core/engine/engine.ts:223-241,263-281`; `player/src/core/scheduler/scheduler.ts:9-14,42-58`
- **Reference:** Checklist §8 async results commit by `{sessionEpoch, queueItemId, attemptId}`; ARCHITECTURE §4b/§5/§11
- **Finding:** Scheduler outcomes carry a stamp, and the playback reducer correctly drops stale events, but `beginResolve` and `prefetchUpcoming` call `setResolution` before checking that the outcome still owns the item's current operation. Cancellation is explicitly best-effort, so a superseded resolver that completes anyway can overwrite a newer success/failure in `QueueItem.resolution`. A built-output probe started two attempts for the same item, completed the new one with `https://fresh`, then completed the canceled old one with `https://stale`: playback remained correctly buffering `https://fresh`, but queue resolution was overwritten to `https://stale`.
- **Why it matters:** The poisoned queue state is the memory cache used by `startItem`. Returning to the item can load the stale/expired bearer URL without a new resolve, creating exactly the “late completion resurrects the wrong URL” failure the stamp invariant exists to prevent. Prefetch/current-attempt races can also replace a valid cache entry or show a false failed state.
- **Suggested fix:** Track the current resolution operation stamp per queue item (or a separate operation registry) and check the full stamp immediately before every queue-state commit. A stale outcome must perform no mutation and no notification. Add same-item old-success-after-new-success, old-failure-after-new-success, current-vs-prefetch supersession, and double-completion tests that assert both playback and `QueueItem.resolution`.
- **Verdict:** CONFIRMED

### [MEDIUM] Appending the first item leaves the queue unable to play

- **Category:** end-to-end-user-value
- **Repo / file / line:** `player/src/core/queue/queue.ts:122-136`; `player/src/core/engine/engine.ts:121-127,169-171`
- **Reference:** no existing invariant — new
- **Finding:** `append` adds IDs to both orders but never initializes `currentItemId`. When the queue is empty (initially, after `setQueue([])`, or after removing its only item), appending a track leaves `currentItemId: null`; `Engine.play()` only starts from a non-null current item and silently does nothing. A built-output probe reproduced the idle, non-playing state with one appended item present.
- **Why it matters:** “Add to queue” is a normal way to move from an empty state to the first playable song. The queue visibly contains the song, yet pressing play cannot start it unless an outer layer knows to issue an extra `selectItem` call. That hidden command-order requirement is easy for future UI/autoplay integrations to miss.
- **Suggested fix:** When appending/inserting into an empty or unselected queue, set `currentItemId` to the first new playable ID (with explicit tests for empty append, insert, remove-last→append, and shuffle state), or make `play()` fall back to `playOrder[0]` and update current atomically.
- **Verdict:** CONFIRMED

## Six-lens summary

- **Technical soundness:** Changes required. Stable IDs, play-order traversal, reducer stamp checks, fallback, and configured SDK boundaries are sound, but queue resolution commits are not stamp-gated.
- **Legal validity:** No finding. A-006's recognized per-item rights policy is present; `stream-legal` remains a fixed source set and does not proxy arbitrary request URLs. No debrid/YouTube/backend implementation exists yet.
- **Overall implementation quality:** All 166 tests/typechecks/builds pass. Tests are deterministic and modules are small, but the advertised race matrix asserts playback only and misses the mutated queue cache; empty-append coverage is absent.
- **System design and operations:** No finding. The player scheduler is memory-only, JIT, retry-free, and separate from a metadata query plane; the current standalone addon deployment caveat is documented and should be re-evaluated when a combined deployment topology lands.
- **UI/UX and accessibility:** No UI is implemented. No separate accessibility finding applies; the empty-queue play dead end and stale-link delay are recorded at the engine layer because they will directly shape future UI behavior.
- **End-to-end user value and delight:** Headless resolve→buffer→play, ranked fallback, pause/seek, auto-advance, and bounded provider failure work. First-item append and stale cached URL reuse make normal queue/replay journeys unpredictable.
