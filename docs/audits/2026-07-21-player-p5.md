## Findings

### [MEDIUM] Debounced queue persistence loses recent state on close or reload

- **Category:** end-to-end-user-value
- **Repo / file / line:** `player/src/ui/viewmodels/usePersistSession.ts:52`
- **Reference:** player ARCHITECTURE §6 and Adversarial Review Contract §2a close/reload journey
- **Finding:** Queue changes are saved only after an 800 ms timer. Cleanup clears the timer without flushing, and there is no `pagehide`/visibility lifecycle flush. Save failures are also detached without recovery or user-visible state.
- **Why it matters:** Closing or reloading within 800 ms of selecting, reordering, or replacing the queue deterministically leaves the previous durable snapshot. On the next launch the app restores stale queue identity, violating the promised restore journey and potentially losing a user's just-made queue edits.
- **Suggested fix:** Keep debouncing normal writes, but serialize/flush the latest snapshot during an appropriate page lifecycle event and on effect cleanup where the host permits it; track the latest queued write and handle persistence rejection. Add a fake-timer lifecycle test reproducing change-then-close-before-800-ms.
- **Verdict:** CONFIRMED

## Resolution (implementer, 2026-07-21)

The finding is accurate and, on investigation, **understated the defect**.

- **[MEDIUM] Debounced queue persistence loses recent state on close or reload
  → FIXED, and a worse variant of it with it.** The report describes an 800 ms
  loss window. The real window was *the entire duration of playback*: the engine
  notifies on every `POSITION` tick (~4×/s from `timeupdate`), and the debounce
  was keyed on *any* notification, so `clearTimeout` fired roughly every 250 ms
  and the trailing write never landed. A user who queued ten tracks and listened
  for an hour had persisted nothing; only pausing for >800 ms flushed it.

  The debounce now lives in `player/src/core/persistence/session-autosave.ts`
  (`SessionAutosave`) — DOM-free, so it is directly testable — and the hook is
  reduced to deciding *when* to feed and flush it. Three changes:
  1. **Only a changed queue reschedules.** `schedule(queue)` is a no-op when the
     snapshot is the one already durable, which makes position ticks free.
     `Engine.getState()`'s referential stability (a P-5 contract) is what makes
     identity a sufficient test.
  2. **Pending snapshots are flushed, not dropped** — on `visibilitychange`
     →hidden, on `pagehide`, and on effect cleanup (`dispose`). Firing on
     `visibilitychange` as well as `pagehide` is what makes this reliable in
     practice; a flush started during teardown is best-effort by nature, and the
     code says so rather than implying a guarantee.
  3. **Rejections are handled.** A failed write keeps its snapshot pending so
     the next edit or flush retries it, instead of silently leaving a stale
     durable copy. It deliberately does **not** retry on its own — a permanently
     failing store (quota, closed database) must not spin. No user-visible
     surface was added: a failed *background queue autosave* has no action for
     the user to take, and inventing an alert for it is scope the slice does not
     need. Writes in flight are serialized, with a newer snapshot picked up when
     the current one settles.

  New tests (`session-autosave.test.ts`, 8): coalescing rapid edits, the
  **starvation** case (40 position ticks × 250 ms → still exactly one write),
  flush-before-debounce-elapses, dispose-flushes-then-stops, retry-after-reject,
  no-spin-on-repeated-failure, newer-snapshot-during-in-flight-write, and
  `markDurable` suppressing an immediate write-back of a just-hydrated queue.

**Verification after the fix:** `player` 180/180 tests (was 172), typecheck, and
`vite build` all green.
